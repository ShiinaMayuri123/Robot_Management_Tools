## 1. Project Overview

**Robot Management Tools** is a web application for tracking and managing robot device information via QR-code scanning or manual MAC address input. Each robot's data (firmware version, APK version, map tool version, etc.) is stored in a cloud database and can be viewed and edited from any device (phone or PC).

Key capabilities:
- **Scan to view**: Scan a QR code attached to a robot → jump directly to that robot's info page.
- **CRUD fields**: Dynamically add, edit, or delete information fields for each robot.
- **Cloud persistence**: Data is saved to Cloudflare KV (production) or SQLite (local dev) via a REST API.
- **Multi-device real-time sync**: The `useRobotSync` hook polls the API every 5 seconds so all open browser tabs/devices stay up-to-date automatically.
- **Toolbox**: Quick-launch buttons for PuduInstaller (via custom `pudu://` URL protocol), external links, and a `.reg` file generator for Windows protocol registration.

#### `src/hooks/useRobotSync.js` ⭐ (Real-Time Sync)
**Role**: Custom React Hook that implements the **5-second polling sync** mechanism.  
**Why it exists**: Separating the polling logic into its own hook keeps `App.jsx` clean and makes the sync behavior reusable and testable independently.

**How it works**:
1. When called with a non-empty `mac`, it immediately fires a GET request to `/api/robot?mac=...`.
2. It then sets up a `setInterval` to repeat this request every **5,000 ms**.
3. The returned `data` object is the raw JSON from the API, and `isSyncing` is `true` while the request is in-flight.
4. When the component unmounts or `mac` changes, the interval is cleared to prevent memory leaks.

**Input**: `mac` (string) — the robot's MAC address  
**Output**: `{ data, isSyncing }` — latest fetched data and sync status flag

**Edit when**:
- Changing the polling interval (currently `5000` ms — search for `setInterval(fetchLatestData, 5000)`)
- Adding WebSocket / Server-Sent Events support as an alternative to polling
- Adding retry logic or exponential backoff on failure

### Backend / API (`functions/`, `server.js`)

---

#### `functions/api/robot.js` ⭐ (Production API)
**Role**: Cloudflare Pages Function — the **production backend API**.  
**Why it exists**: Cloudflare Pages Functions run at the edge (globally distributed). This file handles the two API endpoints that the frontend calls.

**Endpoints**:
| Method | Path | Action |
|--------|------|--------|
| `GET` | `/api/robot?mac=AA:BB:CC` | Reads the robot's data from Cloudflare KV storage |
| `PUT` | `/api/robot?mac=AA:BB:CC` | Writes (upserts) the robot's data into Cloudflare KV |

**Storage**: Cloudflare KV — a globally replicated key-value store. The MAC address is used as the key. The value is the full JSON payload `{ fields: [...], updatedAt: '...' }`.

**Required KV binding**: In your Cloudflare Pages project settings, bind a KV namespace with the variable name **`ROBOT_DATA`**.

**Edit when**: Adding new API endpoints, changing the data schema, adding authentication, or modifying error responses.

---

#### `server.js` ⭐ (Local Development Backend)
**Role**: Local Express.js + SQLite backend server — used **only during local development**.  
**Why it exists**: Cloudflare Pages Functions can't run locally without `wrangler`. This Express server mirrors the same API routes so the frontend can be developed and tested fully offline.

**Storage**: SQLite database at `./db/robot.db` (auto-created on first run).  
**Port**: `3000` (the Vite proxy in `vite.config.js` forwards `/api/*` here).  
**Also includes**: Socket.IO server (currently just logging connections — prepared for future WebSocket-based push updates).

**Edit when**: You want to add new local API routes that mirror Cloudflare Functions, or to implement real-time Socket.IO push events for instant sync (as an alternative to polling).

## 4. Real-Time Sync Architecture

```
Browser Tab A (saves)          Browser Tab B (reads every 5s)
      │                                   │
      │  PUT /api/robot?mac=xxx            │  GET /api/robot?mac=xxx
      ▼                                   ▼
┌─────────────────────────────────────────────────┐
│              Backend API                        │
│  Production: Cloudflare Pages Function + KV     │
│  Development: Express (server.js) + SQLite      │
└─────────────────────────────────────────────────┘
```

**Polling approach** (current):
- `useRobotSync` polls `GET /api/robot` every 5 seconds.
- When new `fields` arrive and the user has no unsaved changes, the UI updates automatically.
- The `isSyncing` flag shows a pulsing green indicator in the header during each request.

**Future upgrade path**:
- Replace polling with WebSocket (Socket.IO is already set up in `server.js`) or Server-Sent Events for instant push from server to all clients.

## 5. Data Flow Diagram

```
URL: /?mac=AA:BB:CC:DD:EE:FF
         │
         ▼
     App.jsx (reads mac from URL)
         │
         ├── fetchData(mac) ──────────────────► GET /api/robot?mac=...
         │       └── sets fields state              └── returns { fields, updatedAt }
         │
         └── useRobotSync(mac) ──┐
                 │               │ every 5s
                 │               ▼
                 │      GET /api/robot?mac=...
                 │         └── syncData updated
                 │
                 └── useEffect([syncData])
                         └── if no unsaved changes → setFields(syncData.fields)
                                                         └── UI re-renders ✅
```

## 6. Running Locally

```bash
# Install dependencies
npm install

# Start both the Vite dev server (port 5173) and Express backend (port 3000)
npm run dev
```

Then open: `http://localhost:5173/?mac=AA:BB:CC:DD:EE:FF`

> **Note**: The SQLite database file `db/robot.db` is created automatically on first run. If the `db/` directory doesn't exist, create it manually: `mkdir db`

## 7. Deploying to Cloudflare Pages

1. Push your code to a GitHub repository.
2. In **Cloudflare Pages**, create a new project and connect your repo.
3. Set the build command to `npm run build` and the output directory to `dist`.
4. Go to **Settings → Functions → KV Namespace Bindings** and bind a KV namespace with variable name `ROBOT_DATA`.
5. Deploy. The `functions/api/robot.js` file will be automatically recognized as a Pages Function.
